## 一、项目概述

### 1.1 产品定位

医学知识库是一个**本地化、单用户**的文档管理平台，用于上传、存储、检索医学类文档（Word / RTF / TXT）。系统核心价值在于**完整保留原始文档格式**（字体颜色、高亮、字号等），并提供跨文档的全文关键词搜索与快速定位能力。

### 1.2 核心用户

医学研究人员、临床医生、医学编辑等需要系统化管理大量文献资料的专业人士。

### 1.3 目标成果

- 文档上传后自动解析入库，无需手动整理
- 保留原始排版和样式，上传即所见即所得
- 积累的文档越多，搜索越有价值，支持全库关键词检索
- 搜索结果直接指向原文关键词位置，降低定位成本

## 二、系统架构

### 2.1 技术选型

| 层级 | 技术方案 | 选型理由 |
|------|---------|---------|
| **前端** | React + TailwindCSS | 组件化开发，生态成熟，搜索交互可控 |
| **后端** | Node.js + Express | 与前端统一语言栈，学习成本低 |
| **数据库** | SQLite + FTS5 | 单用户场景最佳，轻量无需安装，支持全文搜索扩展 |
| **文档解析** | mammoth.js（Word）/ rtlcss（RTF）/ 文本流 | 各格式专用解析器 |
| **富文本渲染** | 原生 HTML 渲染（contenteditable 隔离写入） | 格式零损耗 |

### 2.3 数据流

1. **上传阶段**：用户拖拽/选择文件 → 前端 FormData 上传 → 后端按格式解析 → 内容以 HTML 原始结构存入数据库 → 返回文档 ID
2. **索引阶段**：每次入库时同步写入 FTS5 索引表（文档 ID + 纯文本内容 + 原始 HTML）
3. **搜索阶段**：用户输入关键词 → 后端 FTS5 MATCH 查询 → 返回含关键词片段的文档列表 + 匹配位置
4. **渲染阶段**：点击文档 → 后端返回原始 HTML → 前端渲染 → 搜索关键词在全文内叠加高亮样式

## 三、功能模块详细说明

### 3.1 文档上传模块

#### 支持格式

| 格式 | 文件扩展名 | 解析策略 |
|------|-----------|---------|
| Word 文档 | `.docx` | mammoth.js 解析为 HTML，保留字体颜色、高亮、粗体、斜体、下划线 |
| 富文本格式 | `.rtf` | rtlcss + 自定义解析器转 HTML |
| 纯文本文档 | `.txt` | 直接读取，按 UTF-8 解码 |

#### 格式保留策略

- **字体颜色**：`color` CSS 属性保留
- **背景高亮**：Word 高亮（`shd`）映射为 `<span style="background: #FFFF00">`（黄色高亮）
- **加粗/斜体/下划线**：对应 `<strong>` / `<em>` / `<u>` 标签
- **字号/字体**：保留 `font-size` / `font-family`
- **图片**：mammoth 支持图片提取，以 Base64 内嵌方式存储，保证自包含

#### 上传交互

- 支持**拖拽上传**（Drag & Drop）和**点击选择**两种方式
- 拖拽区域有视觉提示（虚线边框 + 图标 + 文字引导）
- 上传时显示进度条，支持大文件（>10MB）流式上传
- 上传完成后即时返回解析结果，在文档列表顶部显示新增条目并自动展开
- 重复文件名：提示用户是否覆盖或重命名
- 错误处理：格式不支持时提示具体原因（"该文件不是有效的 .docx 文档"）

#### 字段设计

数据库中每条文档记录包含：

```
id          : INTEGER PRIMARY KEY   — 文档唯一 ID
title       : TEXT                  — 文件名（不含扩展名）
file_name   : TEXT                  — 原始完整文件名
format      : TEXT                  — 格式类型（docx/rtf/txt）
html_content: TEXT                  — 原始 HTML 内容（带样式）
plain_text  : TEXT                  — 纯文本（用于全文搜索）
file_size   : INTEGER               — 文件大小（字节）
upload_time : DATETIME              — 上传时间
```

### 3.2 文档列表与阅读模块

#### 列表视图

- 所有已上传文档以**卡片列表**形式展示（垂直滚动）
- 每张卡片显示：文档名、上传时间、格式类型标签
- 支持按上传时间倒序/正序排列
- 支持按格式类型筛选（全部 / Word / RTF / TXT）
- 点击卡片标题 → 在当前页面**展开文档内容**（手风琴式展开）
- 多个文档可同时展开（不限定仅一个）
- 展开后显示原始 HTML 渲染内容（富文本保留）

#### 阅读面板

- 渲染模式：`white-space: pre-wrap; word-wrap: break-word` 确保排版忠实还原
- 文档内禁止编辑（`contenteditable="false"`），防止误改
- 支持打印（`window.print()`）
- 关闭按钮（收起该文档）

### 3.3 全文搜索模块

#### Chrome 风格搜索交互

- 搜索入口位于页面顶部（固定吸顶），始终可见
- 搜索框支持键盘快捷键 `Ctrl/Cmd + F` 聚焦
- 输入关键词后，**500ms 防抖**，自动触发跨文档搜索
- 搜索结果以**列表形式**展示，每个结果包含：
  - 文档名称（点击直接展开该文档）
  - 关键词上下文片段（前后各 30 字，显示关键词高亮）
  - 匹配次数（如"共找到 5 处"）
- 结果列表中点击任意条目 → 展开对应文档 → 页面滚动到该文档位置 → 关键词在全文中高亮
- 支持**下一页/上一页**切换匹配位置（类似 Chrome 的"上一个/下一个"）
- 搜索关键词在全文内以**黄色背景高亮**显示
- 按 `Esc` 关闭搜索面板

#### 搜索结果高亮

- 在文档 HTML 渲染时，遍历全文将关键词替换为 `<mark class="search-highlight">关键词</mark>`
- 高亮样式：`background: #ffe066; color: #333; padding: 0 2px; border-radius: 2px`
- 每次执行新搜索时，先清除旧高亮，再应用新关键词高亮
- 同时在结果列表的摘要片段中也使用相同高亮样式

#### 无结果处理

- 无匹配时显示："未在知识库中找到相关内容，建议尝试更换关键词或上传更多文档"
- 搜索词为空时，显示全部文档列表

## 四、数据存储方案

### 4.1 SQLite + FTS5 全文索引

```sql
-- 主文档表
CREATE TABLE documents (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    title        TEXT NOT NULL,
    file_name    TEXT NOT NULL UNIQUE,
    format       TEXT NOT NULL,
    html_content TEXT NOT NULL,
    plain_text   TEXT NOT NULL,
    file_size    INTEGER,
    upload_time  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- FTS5 全文索引虚拟表
CREATE VIRTUAL TABLE documents_fts USING fts5(
    title,
    plain_text,
    content='documents',
    content_rowid='id'
);

-- 触发器：保持索引与主表同步
CREATE TRIGGER documents_ai AFTER INSERT ON documents BEGIN
    INSERT INTO documents_fts(rowid, title, plain_text)
    VALUES (new.id, new.title, new.plain_text);
END;

CREATE TRIGGER documents_ad AFTER DELETE ON documents BEGIN
    INSERT INTO documents_fts(documents_fts, rowid, title, plain_text)
    VALUES ('delete', old.id, old.title, old.plain_text);
END;

CREATE TRIGGER documents_au AFTER UPDATE ON documents BEGIN
    INSERT INTO documents_fts(documents_fts, rowid, title, plain_text)
    VALUES ('delete', old.id, old.title, old.plain_text);
    INSERT INTO documents_fts(rowid, title, plain_text)
    VALUES (new.id, new.title, new.plain_text);
END;
```

### 4.2 搜索查询示例

```sql
-- 跨文档全文搜索
SELECT id, title, snippet(documents_fts, 1, '<mark>', '</mark>', '…', 32) AS context
FROM documents_fts
WHERE documents_fts MATCH :keyword;
```

## 六、API 接口设计

### 6.1 核心接口

| 方法 | 路径 | 描述 |
|------|------|------|
| `POST` | `/api/documents/upload` | 上传文档，解析入库 |
| `GET` | `/api/documents` | 获取文档列表（支持筛选/排序） |
| `GET` | `/api/documents/:id` | 获取单个文档 HTML 内容 |
| `DELETE` | `/api/documents/:id` | 删除文档 |
| `GET` | `/api/search?q=:keyword` | 全文搜索，返回匹配结果 |

### 6.2 接口详情

#### `POST /api/documents/upload`

**请求**：`multipart/form-data`，字段 `file`（文件）

**响应 200**：
```json
{
  "id": 1,
  "title": "高血压诊疗指南",
  "format": "docx",
  "upload_time": "2026-06-16T10:30:00Z"
}
```

**响应 400**：`{ "error": "不支持的文件格式" }`

#### `GET /api/search?q=高血压`

**响应 200**：
```json
{
  "query": "高血压",
  "total": 3,
  "results": [
    {
      "id": 1,
      "title": "高血压诊疗指南.docx",
      "context": "...<mark>高血压</mark>是常见慢性病...",
      "match_count": 5,
      "upload_time": "2026-06-16T10:30:00Z"
    }
  ]
}
```

## 七、部署方案

### 7.1 本地开发环境

```bash
# 1. 克隆项目后
npm install

# 2. 初始化数据库
npm run db:init

# 3. 启动后端（端口 3001）
npm run server

# 4. 启动前端（端口 3000，代理 API 到 3001）
npm run dev
```

### 7.2 生产环境建议

| 场景 | 推荐方案 |
|------|---------|
| 本地单机使用 | 使用 `nexe` / `pkg` 将 Node.js 打包为可执行文件，双击即用 |
| 内网团队共享 | Docker Compose 部署（前端 Nginx + 后端 + SQLite持久化卷） |
| 云端托管 | Vercel（前端）+ Railway/Render（后端 Node.js）+ SQLite 云数据库 |

## 八、非功能性要求

| 要求 | 说明 |
|------|------|
| **性能** | 单文档 < 5MB 时解析时间 < 3s；搜索响应 < 200ms |
| **数据安全** | SQLite 文件建议定期备份；可考虑加密存储 |
| **浏览器兼容** | Chrome / Firefox / Safari 最新两个版本 |
| **响应式** | 支持桌面端（≥1024px）和平板端（≥768px） |
| **可访问性** | 键盘可操作，搜索框支持屏幕阅读器 |

## 九、后续扩展方向（可选）

以下功能在第一版后可根据用户反馈决定是否实现：

1. **文档标签/分类**：为文档打标签（心内科、糖尿病、影像学等），按标签筛选
2. **文档版本管理**：同一文档重新上传时保留历史版本
3. **导出功能**：将搜索结果文档批量导出为 PDF
4. **AI 问答**：基于文档内容进行自然语言问答（接 LLM API）
5. **移动端适配**：支持手机端浏览（优先级低）
